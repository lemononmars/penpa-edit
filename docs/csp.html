Sudoku CSP architecture

The Auto Solver and full solution search use the same constraint-satisfaction problem runtime in js/sudoku_csp.js. Its architecture follows cspuz: puzzle code declares constraints and answer variables, while one reusable engine performs search and answer analysis.

Problem model

CSP conceptPenpa+ representation
VariablesThe 81 cells in a classic 9×9 Sudoku.
DomainsDigits 1–9, filtered by assigned values and registered constraints.
Core constraintsAll-different rows, columns, and 3×3 boxes.
Variant constraintsPlug-ins registered by a stable string key such as kropki.
Answer keysAll 81 cell variables. Helper variables added by future variants do not need to be exposed as answers.
SearchBacktracking with minimum-remaining-values selection and forward consistency checks. The engine is currently pure JavaScript rather than SAT-encoded.

Candidate overlay

Candidate marks are global, not merely local eliminations. For each remaining cell/digit pair, the engine finds a complete solution containing that assignment. A digit is shown exactly when such a witness exists. A one-value domain is therefore an irrefutable fact: it has that value in every solution.

Problem API

var problem = SudokuCSP.createProblem(board, constraints);
problem.solve();                 // first satisfying assignment
problem.enumerateAnswers(10);    // bounded answer enumeration
problem.irrefutableFacts();      // exact candidates and forced values

irrefutableFacts() returns satisfiable, unique, candidates, and forced. The editor caches this analysis until the board or its normalized constraints change. Redraws caused only by toolbar or mode clicks reuse the completed result without running the solver again.

Incremental reruns

The asynchronous extractor retains the complete-solution witnesses from its previous run. After a digit is added or changed, it filters those witnesses against the current givens and reuses every compatible solution to seed the next exact extraction. Constraint changes use a different signature and therefore start without stale witnesses. Reuse is an optimization only: every displayed candidate is still backed by a complete solution under the current board and constraints.

Each automatic analysis and generation run has a hard 60-second deadline. The cooperative CSP search stops at its next cancellation checkpoint, reports the limit in the solver log, and leaves the Penpa board unchanged.

Why this does not enumerate every solution

Enumerating every completed grid can be prohibitively expensive and is unnecessary for exact pencil marks. The extractor starts with one or more complete-solution witnesses, then tests each not-yet-witnessed cell/digit assignment. A successful test contributes another complete-solution witness; a failed test proves that answer fact impossible. This produces the same union of candidate values as full enumeration while usually visiting far fewer solutions. The on-page CSP log shows the start, terminal result, and total runtime while Auto Solver runs.

Unique puzzle generation

The generator first constructs a completed grid satisfying the currently active supported variants. It derives valid Odd/Even, Kropki, XV, and Battenburg marks from that answer; every mark is created together with its 180-degree rotated partner. It then visits every rotational pair of given digits and generated marks in random order and temporarily removes both members. After every proposed pair removal it calls problem.enumerateAnswers(2): one answer keeps the removal, while zero or two answers restore both clues. Because clue removal is monotonic, one complete pass leaves a pair-minimal puzzle: no remaining rotational clue unit can be removed while retaining uniqueness. Global Diagonal, Anti-Diagonal, Anti-King, Anti-Knight, and Non-Consecutive constraints compose with those generated marks. Generation runs in a Web Worker so repeated searches do not block Penpa's canvas.

New grid constructs that completed answer from scratch. From existing first asks the CSP to complete the current givens and semantic clues, adds compatible generated marks, and then runs the same symmetric minimal-pruning pass. A generated puzzle also primes the candidate cache with its already-proven unique answer, so opening Auto Solver does not repeat the expensive initial Anti-Diagonal search.

Adding a constraint

SudokuCSP.registerConstraint("myVariant", {
    validatePartial: function(board, constraint, helpers) {
        var value = helpers.cellValue(board, constraint.cell);
        return !value || constraint.allowed.includes(value);
    }
});

Then normalize Penpa editor objects into constraints.myVariant in sudoku_solver.js. The candidate overlay and search automatically invoke the new handler.

Registered today

  • Classic Sudoku: core row, column, and box constraints
  • Odd / Even: grey circles require odd digits and grey squares require even digits
  • Diagonal: both long diagonals are all-different
  • Anti-Diagonal: both long diagonals contain exactly three digits repeated three times each
  • Anti-King: equal digits cannot be a king's move apart
  • Anti-Knight: equal digits cannot be a knight's move apart
  • Non-Consecutive: orthogonal neighbors cannot differ by one
  • Thermo: strictly increasing paths
  • Arrow: circle equals shaft sum
  • Killer: distinct cage digits with a target sum
  • Kropki: black 1:2 dots and white consecutive dots, with an optional Negative Kropki toggle for undotted edges (off by default)
  • Palindrome: mirrored cells along each line must contain equal digits
  • XV: V edges sum to 5 and X edges sum to 10, with an optional negative rule for unmarked edges (off by default)
